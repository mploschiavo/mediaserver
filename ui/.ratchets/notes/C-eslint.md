# C — ESLint ratchet rules

Three rules are locked at 0 occurrences in `ui/eslint.config.js` and enforced by `pnpm lint` (which already passes `--max-warnings=0`): `no-console` rejects any direct `console.log` / `console.debug` / `console.info` call in production source under `src/**/*.{ts,tsx}` (it allows `console.warn` and `console.error` so genuine error-handling paths keep working); `@typescript-eslint/no-explicit-any` blocks new `: any` annotations and casts so we keep the type surface honest; and `no-only-tests/no-only-tests` prevents `.only(` from sneaking into committed tests (which would silently skip the rest of the suite). The rules are scoped so the OpenAPI-generated `src/api/types.ts` and the router-generated `src/routeTree.ts` are ignored, and `no-console` is intentionally *not* applied to test files (`*.test.{ts,tsx}`, `*.spec.{ts,tsx}`, `__tests__/**`, `src/test/**`) — the other two ratchets still apply there. Escape hatches exist for genuinely justified cases: use `// eslint-disable-next-line no-console -- justified because we need raw stdout for the CLI bootstrap`, `// eslint-disable-next-line @typescript-eslint/no-explicit-any -- justified because this third-party type is truly unconstrained`, or `// eslint-disable-next-line no-only-tests/no-only-tests -- justified because <reason>`. Always include the `-- justified because …` comment so reviewers see the rationale; bare disables should be rejected in code review.
