# Orchestrator coverage matrix

Cross-reference of legacy auto-heal cycle hooks vs ADR-0003
orchestrator coverage. Used to scope Phase 5d (retiring legacy hooks)
safely. Updated 2026-05-02 from live shadow + live primary data.

## Legacy auto-heal hooks (current state)

The 60s tick in `api/services/auto_heal.py` runs four `run_job(...)`
calls:

| Legacy hook | Orchestrator coverage | 5d action |
|-------------|----------------------|-----------|
| `guardrails:evaluate` | None — guardrails are a separate evaluation pipeline (state-snapshot + threshold rules), not promise-driven | **Keep** |
| `jellyfin:ensure-api-key` | `jellyfin-api-key-discoverable` promise → `JellyfinLifecycle.mint_api_key` → calls `http_preflight.run_preflight` (same impl) | **Retire candidate** — equivalence verified |
| `jobs:close-stale-runs` | None — framework maintenance (closing zombie `status=running` records); not a service lifecycle | **Keep** |
| `orchestrator:satisfy-shadow` | This IS the orchestrator | Keep (rename when out of shadow) |

So Phase 5d only retires **one** hook: `jellyfin:ensure-api-key`.

## Orchestrator-driven primary mode (Phase 5c live)

`ORCHESTRATOR_LIVE_SERVICES` includes all 16 services with a
`ServiceLifecycle` implementation. Per-service ensurer dispatch:

### LifecycleEnsurer (orchestrator calls a Protocol method)

| Service | Method invoked | Underlying impl |
|---------|---------------|-----------------|
| jellyfin | `mint_api_key` | wraps `infrastructure.jellyfin.http_preflight.run_preflight` |
| sonarr/radarr/lidarr/readarr/prowlarr | `mint_api_key` | poll-and-wait for `<svc>/config.xml` `<ApiKey>` (auto-generated by *arr) |

### JobEnsurer (orchestrator dispatches a JobRunner job by name)

These are existing `ensure-*` jobs the orchestrator invokes when
the corresponding promise's probe fails. The orchestrator is a
SCHEDULER for these; the implementations are the same code that the
legacy bootstrap pipeline runs.

## Negative test results (live, 2026-05-02 compose)

Three tests run, breaking each promise's invariant and watching
the orchestrator restore it. Auto-heal cycle was running in
parallel for all three, but only the orchestrator-dispatched jobs
heal these specific invariants — so attribution is clean for
TEST 1 and TEST 2.

### TEST 1: radarr-import-lists-auto (clean attribution)

```
23:04:58  flipped enableAuto=False on TMDb Trending + Upcoming
          (legacy auto-heal does NOT touch import lists)
23:05:52  orchestrator: ensurer fired: radarr-import-lists-auto (kind=job)
23:05:52  patch_arr_import_lists_auto: 'TMDb Trending Movies' False → True
23:05:54  patch_arr_import_lists_auto: 'TMDb Upcoming Movies' False → True
end       all 4 lists at enableAuto=True
```

**~54 seconds end-to-end self-heal.** Only the orchestrator can have
done this — auto-heal cycle's hooks don't dispatch
`apply-arr-runtime-defaults`. Proves `JobEnsurer` path works.

### TEST 2: qbittorrent-categories (clean attribution)

```
23:30:10  removed 'movies' category via qbit API
          (legacy auto-heal does NOT call ensure-qbittorrent-categories)
23:30:46  orchestrator: ensurer fired: qbittorrent-categories (kind=job)
23:30:47  ensure-qbittorrent-categories ran
end       categories restored: ['books, movies, music, readarr, tv']
```

**~37 seconds.** Note: the ensurer logs `SKIPPED — ['tv', 'music',
'books']` which lists categories it kept — confusing message but
the world IS healed (movies recreated). The silent-error-as-ok
shape of this ensurer is tracked separately in
`bug_class_silent_error_as_ok.md` memory.

### TEST 3: jellyfin api key (functional pass, muddier attribution)

```
03:13:06  deleted all rows from jellyfin's ApiKeys SQLite table
03:13:57  auto-heal jellyfin:ensure-api-key: SKIPPED — already_minted
          (env still cached previous key value)
03:14:03  orchestrator dispatched ensure-jellyfin-libraries
          (different promise; runs through http_preflight on auth-fail)
03:14:09  http_preflight minted new key
end       DB has 1 key
```

The proximate-cause action was the orchestrator dispatching
`ensure-jellyfin-libraries` (a JobEnsurer) which internally hit a
401 and re-minted via `http_preflight.run_preflight`. The
LifecycleEnsurer path (`JellyfinLifecycle.mint_api_key`) was
NOT exclusively responsible for this heal — both pipelines
converged on the same `run_preflight` call. Functionally, the
heal worked; cleanly attributing it to the orchestrator's
LifecycleEnsurer alone would have required disabling both the
legacy `jellyfin:ensure-api-key` auto-heal hook AND the
`jellyfin-libraries` JobEnsurer first.

## Phase 5d safety

Based on TEST 1+2: the orchestrator's `JobEnsurer` dispatch is
proven to fire and heal at the right time, on the right cadence,
through the existing JobRunner infrastructure.

For `jellyfin:ensure-api-key` specifically (the one Phase 5d
retires): the orchestrator's `LifecycleEnsurer` invokes the SAME
`run_preflight` function the legacy hook does. Equivalence is by
construction, not by side-by-side measurement. Phase 5d retiring
this hook therefore reduces redundant calls (one per minute
instead of two) without changing functional coverage.

## Untested but likely-fine surface

The 25-ish other `ensure-*` JobEnsurers in the registry haven't
been individually negative-tested. They follow the same dispatch
shape as `apply-arr-runtime-defaults` and `ensure-qbittorrent-
categories` — same dispatcher, same JobRunner, same idempotent
contract. The risk of a class-wide bug is low; the risk of
per-handler bugs is service-by-service.

If a per-handler bug exists, the orchestrator would surface it
the same way it surfaced the silent-error-as-ok shapes during
Phase 4d (`ensure-qbittorrent-categories` returning ok with
`err=login failed`, `ensure-bazarr-language-profile` returning ok
with `settings POST 500`). Those are tracked in
`bug_class_silent_error_as_ok.md` memory.

## Recommendation

Phase 5d is safe to do for `jellyfin:ensure-api-key`. It's NOT
safe to retire `guardrails:evaluate` or `jobs:close-stale-runs`
(no orchestrator coverage). Phase 5e (deleting the legacy
preflight code paths) needs a separate audit per service-family,
not a blanket pass.
